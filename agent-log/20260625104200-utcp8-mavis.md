## 对话与行动记录

### 1. 用户反馈的核心问题

旧实现流程：
1. 探测所有安装的 provider
2. 用 `withTaskGroup` 并发跑所有 pipeline
3. **等所有完成** → `await for ... in group` 退出循环
4. 一次性把全部结果合并后赋给 `state`（`@Published`）
5. SwiftUI 在 `@ObservedObject` 绑定下响应 state 变化（v0.3.0-UI-A-004 已实现）

结果：**虽然 SwiftUI 能响应 state 变化，但 state 本身只在所有 pipeline 都完成时才更新一次**。用户看到的体验是"白屏 → 一瞬间全部数据出现"，中间没有任何 provider 完成时的中间态。即使不关 dropdown，新数据也是在所有 provider 都完成之后才一次性出现，跟"白屏 → 一次性出现"没区别。

而且菜单栏 bar 也只在所有快照替换完成时才一次性重画，没有"逐步增加"的视觉。

### 2. 设计：streaming refresh

新流程：

```
T0:   detectInstallReasons() → [codex: "...", minimax: "...", ...]
T0+ε: 立刻 seed placeholder：
       state.snapshots = [
         .loading(codex),
         .loading(minimax),
         .loading(kimi),
       ]
       按用户偏好顺序排好
       refreshState = .refreshing
       → UI 立刻看到 3 个 provider 行 + 骨架
       → 菜单栏立刻看到 3 个 dimmed 50% bar
T1:   codex pipeline 完成 → applyProviderResult
       → state.snapshots[0] = real .available(codex)
       → UI 第 1 行从骨架变为真实数据，菜单栏第 1 个 bar 从 dimmed 变全亮
T2:   minimax pipeline 完成 → applyProviderResult
       → state.snapshots[1] = real .available(minimax) with quotas
       → UI 第 2 行从骨架变为真实数据，菜单栏第 2 个 bar 高度跳变
T3:   kimi pipeline 完成 → applyProviderResult
       → state.snapshots[2] = real .available(kimi)
       → UI 第 3 行从骨架变为真实数据
T4:   所有 child task 退出 → 收尾：
       refreshState = .succeeded
       needsFullDiskAccess 重新计算
```

### 3. 关键改动

#### 数据层（`QuotaModels.swift`）

- `ProviderAvailability` 新增 `.loading` case（注释强调它是「探测到已装 + pipeline 尚未返回」的过渡状态）。
- `ProviderSnapshot.loading(kind:fetchedAt:)` 静态工厂：availability=.loading、quotas=[]、tier/price=nil、`isStale=false`。
- `statusColor(itemOrder:)` 加 `.loading` 分支：返回灰色 `#8E8E93`，与 needsConfiguration 同色但语义独立。

#### 协调器（`RefreshCoordinator.swift`）

`runRefreshCycle()` 完全重写为 streaming：

1. `detectInstallReasons()` 找出所有 active kinds（installed + 未隐藏）。
2. **立即** build `ProviderSnapshot.loading(kind:)` 占位 list，按用户 `providerOrder` 排好，写入 `state` + `refreshState = .refreshing`。这一步不需要 await，是同步赋值，所以 UI 立刻拿到新 state（SwiftUI 在下一个 runloop tick 渲染）。
3. `withTaskGroup`：每个 active provider 一个 child task，跑 `fetchSnapshot`，**完成时回到 MainActor 立即调 `applyProviderResult`**。每个 provider 完成都是一次独立的 `@Published` 触发。
4. 监听 `for await _ in group` 仅用于等所有 child 完成，**不**在中间汇总 state（避免重复 publish）。
5. 所有 child 完成后做一次收尾：计算 `refreshState`（succeeded / partialFailure / failed）、重算 `needsFullDiskAccess`、最终 publish 一次 state。

`applyProviderResult(kind:result:installDetail:now:)` 新方法：

- `result.success(snapshot)` → 用 `pickBestSnapshot` 选最优（单 strategy 通常直接返回该 snapshot）。
- `result.failure(error)` → 按错误类型生成 fallback：`needsConfiguration(reason: installDetail)`（如果探测到装）/ `error.availabilityFallback`（QuotaFetchError 自带映射）/ `.fetchFailed(reason:)`（其他错误）。
- 在 `state.snapshots` 里找该 kind 的位置（同位置替换 placeholder，保持 UI 顺序稳定）；fetchFailed / notInstalled 直接移除条目（与原行为一致）。
- `state.lastUpdated = max(prev, snapshot.fetchedAt)`：每个 provider 完成都推进 lastUpdated，让顶部"上次更新 HH:mm"逐 provider 前进。

新增私有静态方法 `sortByProviderOrder(_:_:order:)`：避免与 `applyProviderOrder`（订阅排序变更时使用）的逻辑重复。

`pickBestSnapshot` 的 availability 优先级调整：`.loading` 加进去（priority=2，介于 needsConfiguration=3 和 notInstalled=1 之间）。这主要是为了满足 exhaustive switch warning，loading snapshot 实际上不会进 pickBestSnapshot（applyProviderResult 拿到的是真实 result）。

#### 状态栏（`StatusBarController.swift`）

- `drawableSnapshots` 加 `.loading` → true（显示为 dimmed 50% bar）。
- `remainingFraction` 加 `.loading` → 0.5。
- `makeBarsImage` 填充时检查 `.loading` → `NSColor(white: 1.0, alpha: 0.4)`（其他 bar 用 alpha 1.0）。视觉效果：dimmed 占位 bar 让用户看到"这个 provider 还在刷新"，同时保持可见以体现"动态增长"。
- tooltip：loading 的 provider 显示「刷新中」而不是 50%，避免误导。

#### dropdown（`MenuView.swift`）

- `PlanSection.body` 加 `case .loading: QuotaSkeleton()`：loading snapshot 直接走骨架分支，不需要依赖 `isLoading` 参数。
- `ReadyStateView` 把 `isLoading: false`（硬编码）改为 `isLoading: snapshot.availability == .loading`（派生），per-snapshot 独立判断。

`LoadingStateView` 保持现状：所有 snapshot 都是 `.loading` 时仍然全骨架（`isInitialLoading` 走 `LoadingStateView` 分支，因为 `.loading != .available` → `hasAnyAvailable = false`）。代码可以清理（`isLoading: true` 改为派生），但当前形态也正确，先不动。

#### Pipeline 优先级（`ProviderFetchStrategy.swift`）

- `FetchPipeline.merge` 的 priority closure 加 `.loading` case：priority=2。同样主要是为了 exhaustive switch。

### 4. 与现有逻辑的兼容性

- **隐藏 provider**：`activeProviders` 过滤 `!hiddenKinds.contains($0.kind)`，hidden kinds 不进入 placeholder 列表，自然不显示。
- **未安装 provider**：探测阶段就过滤掉，根本不会 seed placeholder，state.snapshots 里也没它们。
- **重新刷新期间**：state.snapshots 会被一次性替换为新的 `.loading` placeholder list，**所有之前的数据短暂消失**（替换瞬间）→ 立刻被各 child 完成后的真实数据回填。可见的 flicker 是：真实数据 → 骨架 → 真实数据流。如果想消除，需要加 `isRefreshingNow: Bool` 字段保留旧数据，但当前用户没要求，先不做。
- **首启动**：所有 active kind 都是 `.loading` → `LoadingStateView` 显示 → 第一个 provider 完成后切到 `ReadyStateView` → spinner 在 summary header 持续显示直到所有完成。
- **dashboard state 完整性**：`applyProviderResult` 始终保证 state.snapshots 的合法性（fetchFailed / notInstalled 会被剔除）。
- **subscription group 顺序**：`primarySubscriptionGroupWorstQuota` 仍然按用户订阅组顺序取值；streaming 不改变这条逻辑，只是数据到达更快。
- **PreferencesStore 不变**：streaming 是 coordinator 内部行为，不涉及持久化。

## 备注

- **隐藏 / 重刷新 flicker**：当前重刷新瞬间所有 snapshot 会从真实数据 → 骨架 → 真实数据（依次回填）。如果用户觉得这个 flicker 不舒服，再加 `isRefreshingNow` 字段保留旧数据。本轮先按用户原话「先显示刷新中，等获取到再显示具体的值」实现。
- **`primarySubscriptionGroupWorstQuota` 不变**：streaming 只改 coordinator 发布时机，不改 quota 取值逻辑。
- **fetchFailed 的隐藏时机**：applyProviderResult 里把 fetchFailed snapshot 从 state.snapshots 移除，但仅当 replaceAt 成功（state 里确实有 placeholder）。如果 placeholder 已被某种异常路径移除，applyProviderResult 会 fallback 走 `if let index = ... else if keepAfterApply`，不会重复插入。
- **state.lastUpdated 在 hide()/clearHidden() 时的语义**：hide() 现在直接清空 state.snapshots 不会动 lastUpdated；如果用户 hide 掉所有 provider，lastUpdated 仍保留上一次刷新时间。这是合理的——「我最后一次成功刷新是什么时候」与「当前可见几个 provider」是两件事。
- **v0.3.0-UI-A-004 是 streaming 的前提**：没有 @ObservedObject 绑定 coordinator，per-provider publish 不会触发 UI 更新。当前这套已就位，所以 streaming 直接受益。
