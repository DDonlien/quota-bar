### 根因分析

user 报"codex 已过期但刷出月额度"——直观看是 parser 误把 free 用户的"月额度"当 Plus 5h 窗口处理。

```bash

```

`sub/main` 基于 `deee18e` (main HEAD)，working tree clean。

**Step 2：跟 user 拿实际数据**

通过 ask_user 让 user 选 Codex 过期状态响应格式，user 回复"你自己研究呀？思路还是.app授权信息/cli指令/web抓取这个顺序"——这给了一个**重要信号**：

> 订阅状态最权威的来源 = 服务商在用户登录时写下来的本地授权元信息（不是从 API quota 数值反推）

按 user 提的优先级："app 授权信息 > CLI 指令 > Web 抓取"。

**Step 3：挖 ground truth**

- 看 user 的 `~/.codex/auth.json`（已装 Codex.app，但没装 `codex` CLI）
- `auth.json` 里的 `id_token` 是个 RS256 JWT，payload 里有：
  ```json
  "https://api.openai.com/auth": {
    "chatgpt_plan_type": "plus",
    "chatgpt_subscription_active_start": "2026-05-25T15:23:47+00:00",
    "chatgpt_subscription_active_until": "2026-06-25T15:23:58+00:00",
    ...
  }
  ```
  → 过期 3.5 天（当前 2026-06-29）。这正好印证 user 思路里的 ".app 授权信息"。

- 用 refresh_token 调 `https://auth.openai.com/oauth/token` 拿到新 access_token，调 `wham/usage` 拿真实响应：
  ```json
  {
    "plan_type": "free",                       ← 关键
    "rate_limit": {
      "primary_window": {
        "used_percent": 5,
        "limit_window_seconds": 2592000,        ← 30 天 = "月额度" 的来源
        "reset_at": 1785288790
      },
      "secondary_window": null                  ← 周窗口消失
    }
  }
  ```
  → 现有 `CodexDashboardParser.parse` 在 `pickWindow(names: ["five_hour", "primary_window", ...])` 找到 "primary_window" → periodSeconds=2592000 → UI 推断成"月额度" → 显示"5% 月额度"——**完全误导**。同时 `parseTier` 解析 `plan_type: "free"` 但没用来给 snapshot 标"过期"。

**核心 bug：**
1. parser 不知道 `plan_type=free` 是"订阅过期"信号
2. 整个 pipeline 没有"权威订阅状态"检查，只能从 quota 数值反推（永远不可靠）

### 方案设计

按 user 提的优先级，**v0.8.0 第一批只做 Codex**（user 当前 case），其他 provider 沿用 inspector 模式扩展（v0.8.0-DATA-B 登记 deferred）。

**核心思路**（"权威 > API 反推"模型）：

1. **Level 1（最权威）**：解析 `id_token` 的 `chatgpt_subscription_active_until` < now → 100% 过期（不依赖网络）
2. **Level 2（次权威）**：`plan_type == "free"` → 判定过期
3. **Level 3（兜底）**：`wham/usage` 响应里 `secondary_window == null` 或 `primary_window.limit_window_seconds` 从 18000 跳到 2592000 → 暗示降级（用 `parseSubscriptionExpiresAt` / `parseSubscriptionStatus` 解析）

**实现架构：**

- 新增 `SubscriptionInspector.swift`：
  - `JWTPayloadDecoder`：解 JWT payload（不验签）
  - `OpenAIAuthPayload` struct：把 `https://api.openai.com/auth` 命名空间字段组装成 Swift 类型
  - `SubscriptionStatus` 枚举：`.active / .expired / .free / .unknown`
  - `CodexSubscriptionInspector`：从 `~/.codex/auth.json` 解 id_token → 映射成 `SubscriptionStatus`
- `ProviderAvailability` 加 `.subscriptionExpired(plan, expiredAt)` case
- `QuotaFetchError` 加 `.subscriptionExpired(plan, expiredAt)` case
- `CodexAuthProvider.fetchSnapshot`：先调 inspector，过期时**直接返回 marker snapshot**（不 throw！让 pipeline 立即返回，避免后续 strategy 用 free 月度窗口覆盖）
- `CodexDashboardParser`：`plan_type == "free"` 时返回 nil（防御 BrowserCookie 路径）
- `BrowserCookieProvider`（Codex 路径）：也调 inspector，与 CodexAuthProvider 行为一致
- `MenuView.PlanSection`：`.subscriptionExpired` 渲染灰标 hint "订阅已过期 · Plus · 到期 2026/6/25"
- `StatusBarController`：`.subscriptionExpired` 仍画 0% 最小 bar，tooltip 显示"已过期"
- `RefreshCoordinator.applyProviderResult.keepAfterApply`：`.subscriptionExpired` 当 keep=true
- 39 个新单元测试覆盖 JWT 解析 / inspector 6 种路径 / parser free 路径

**关键设计决策：**

- **返回 snapshot 而不是 throw**：throw 会让 pipeline 走到下一个 strategy，CLILogProvider 仍然会显示 5h/weekly 误数据。**返回** marker snapshot 让 pipeline 立即返回，避免误导。
- **不验证 JWT 签名**：应用已持有完整 token 字符串，验签需要后端 JWKS；即使伪造也只能误导本机 UI（攻击者已有 root）；ChatGPT 后端在 API 调用时仍会验签。
- **失败统一返回 .unknown，不要瞎猜**：文件缺失、字段缺失、JWT 坏等情况都返回 .unknown，**不**用"没看到 expired 字段就是 active"反推。
