     - Code 5小时额度（`ratelimitCode5h.enabled` → 有 ratio 则计算，否则 100%）

## 追加修复 2（MiniMax API Key 输入无法输入）

**问题根因**：NSMenu 的 `trackMouse` 事件循环与 SwiftUI 的 `TextField` 在 `NSHostingView` 中的焦点机制不兼容。SwiftUI `TextField` 依赖的事件链在 NSMenu 的 custom view 中被截断，导致用户点击后无法获取焦点、无法输入。

**解决方案**：用 AppKit `NSTextField` 的 `NSViewRepresentable` 包装替换 SwiftUI 的 `TextField`：
- 新建 `APIKeyTextField`（`NSViewRepresentable`），底层是 `NSTextField`
- 保留 `.roundedBezel` 样式、`controlSize: .small`、等宽字体
- `NSTextField` 作为原生 AppKit 控件，不受 NSMenu trackMouse 循环的 SwiftUI 间接层影响，可直接接收焦点和键盘输入

文件：`MenuView.swift`

构建产物：`quota-bar/build/20260621-025759/QuotaBar.app`

## 追加修复 4（quota 分层获取机制）

用户要求实现三层获取机制：
1. **App 直接获取**（OAuth/API）
2. **CLI 指令获取**
3. **Headless 浏览器**（Edge Cookie）

### 实现

#### 1. MiniMax CLI 获取额度

新建 `MiniMaxCLIProvider.swift`：
- 执行 `/opt/homebrew/bin/mmx quota --output json --no-color`
- 解析 JSON 中的 `model_remains` 数组
- 每个 model 生成一个 `QuotaWindow`（weekly 剩余百分比作为额度）

#### 2. Edge Cookie 读取支持

新建 `EdgeCookieReader.swift`：
- 直接读取 `~/Library/Application Support/Microsoft Edge/Default/Cookies` SQLite 数据库
- 复制到临时位置后解析，避免原文件锁定
- 返回 `HTTPCookie` 数组，供 `BrowserCookieProvider` 使用

已验证 Edge Cookie 数据库中存在 `kimi-auth` cookie（用户已登录 Kimi）。

#### 3. Pipeline 优先级更新

`Strategies.swift` 更新后的优先级：

**Kimi**：
1. `BrowserCookieProvider` (Edge Cookie) — 获取 Work + Code 全部 3 个额度
2. `BrowserCookieProvider` (Chrome Cookie) — 兜底
3. `KimiAuthProvider` (OAuth) — 获取 Code 额度
4. `KeychainProvider` — 最终兜底

**MiniMax**：
1. `MiniMaxCLIProvider` (mmx) — CLI 获取额度
2. `BrowserCookieProvider` (Edge Cookie) — web 已登录
3. `BrowserCookieProvider` (Chrome Cookie) — 兜底
4. `MiniMaxConfigProvider` (~/.mavis/config.yaml API Key) — 配置文件
5. `KeychainProvider` — 最终兜底

**Claude**：
1. `BrowserCookieProvider` (Edge Cookie)
2. `BrowserCookieProvider` (Chrome Cookie)
3. `KeychainProvider`

**Antigravity**：
1. `AntigravityDashboardProvider` (language_server)
2. `KeychainProvider`
