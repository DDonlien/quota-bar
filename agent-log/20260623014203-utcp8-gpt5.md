> 
> 首先，同一个 Agent 下的子服务应该顺序排列。比如说 MiniMax 它有 General 和 Video，那它就应该是：
> 1. General 5小时
> 2. General 周
> 3. Video 日
> 4. Video 周
> 
> 而不是把两个小额度凑到一起。
> 
> 其次，如果一个 agent 存在超过一个可用的对象（比如说 Kimi 有 code 和 work），那这个小黄灯（也就是模型或服务前面的那个灯）应该优先显示顺序更靠前的那个.菜单栏显示的 bar 也是同理。但这里的显示顺序要加一个功能，允许长按住之后拖拽。当然，这个拖拽是整组拖拽，并且不能离开这个模型的范围。通过拖拽来上下调整
> 
> 但这里只关心最小时间颗粒度的额度：
> 1. 比如 work 只有月额度，那就看月额度
> 2. code 有 5 小时和周额度，那就只看 5 小时额度

### 1. `ProviderSnapshot` 新增 `sortedQuotas`（框架级别统一排序）

**`QuotaModels.swift`**：

- 例如 MiniMax：General 组内 5h → 周，Video 组内 日 → 周，整体顺序：General 5h → General 周 → Video 日 → Video 周

### 2. 指示灯（`statusColor`）取第一个 quota

- 例如 Kimi：`Code 5h` 排在第一，如果 Code 5h 100% 但 Work 月 17%，灯显示绿色（只看 5h）

### 3. MenuView 使用统一排序

### 4. 菜单栏 Bar 取第一个 quota

**`StatusBarController.swift`**：

- 不再考虑 `resetsAt` 的时效性，只取最小时间颗粒度
- 移除 20 行复杂的 `upcoming` / `dated` / `periodSeconds` 回退逻辑

### 5. 关于拖拽排序

用户要求"长按住之后拖拽，整组拖拽，不跨 agent 范围"。

**技术限制**：当前架构是 `NSMenu` + `SwiftUI NSHostingView`。`NSMenu` 的 `trackMouse` 循环会拦截所有鼠标事件，SwiftUI 的 `DragGesture` 无法正常工作。在 NSMenu 的 custom view 中实现整组拖拽需要：
1. 重写 AppKit 级别的鼠标事件处理（绕过 NSMenu 的 trackMouse）
2. 手写拖拽状态管理、动画、位置重排
3. 重建整个 Menu 的 view 层级

**建议**：如果拖拽排序是强需求，需要更大的架构改动（如改用 `NSPopover` 替代 `NSMenu`，或使用 AppKit 原生实现）。当前建议用按钮上下移动（↑/↓）作为替代方案。
