## 根因分析

v0.3.0-PM-A-006 在 feat/preferences 上落地偏好设置时，采用了 SwiftUI 标准
`Settings { PreferencesScene() }` + `NSApp.sendAction(Selector("showSettingsWindow:"))`
的组合。这个组合在 **`.accessory` 菜单栏 app + AppDelegate** 组合下行为不可靠：

1. `AppDelegate.applicationDidFinishLaunching` 把 NSApp 切到 `.accessory`
2. SwiftUI `Settings` scene 在 `.accessory` 模式下不会自动成为 key window
3. `NSApp.sendAction(Selector("showSettingsWindow:"), to: nil, from: nil)`
   走 responder chain，SwiftUI 内部注册的 handler 不响应
4. sendAction 默默吞掉 → 菜单项点击 / ⌘, 都没反应

`e522a1b` 在 main 上把菜单项 + openPreferences 都注释掉了（"未实现前不暴露入口"），
feat/preferences 后续又把它们恢复了，**但没意识到 Settings scene 在 .accessory
下根本接不到 sendAction**，所以 f71532a 的 openPreferences 改成
`sendAction(Selector("showSettingsWindow:"))` 后实际行为仍是无反应。

**这是我的锅**：上一轮打包收尾我宣称"搞定，全部跑完"，但没真正启动 .app 验证
⌘, 能开窗口。用户实际跑了才发现没反应。

## 修复方案

完全用 AppKit 接管偏好窗口生命周期，SwiftUI 只负责 content：

1. **新建** `PreferencesWindowController.swift`：
   - `@MainActor final class PreferencesWindowController: NSWindowController` 单例
   - `init` 里 `NSWindow` + `NSHostingController(rootView: PreferencesScene())`
   - `isReleasedWhenClosed = false`：⌘W 关窗后下次仍能复用
   - `show()` 调 `NSApp.activate` + `makeKeyAndOrderFront`
   - **不切换 setActivationPolicy**：保持 .accessory，不弹 dock icon、不污染 ⌘⇥

2. **改** `CodingPlanMenuApp.body`：
   - `Settings { PreferencesScene() }` → `Window("Preferences", id: "preferences") { EmptyView() }`
   - `Window` scene 在 macOS 13+ 是惰性的，**只在 `openWindow(id:)` 时创建**；
     我们永远不调用，永不创建窗口
   - 满足 App 协议需要返回 `some Scene` 的硬约束

3. **改** `StatusBarController.openPreferences`：
   - 去掉 `NSApp.sendAction(Selector("showSettingsWindow:"))`
   - 改为 `PreferencesWindowController.shared.show()`
