## 完成工作

### 1. 新增 `.github/workflows/release.yml`

一个 workflow 两个 job：

- **`build`**：每次 push main + PR + 手动触发都跑
  - checkout + SwiftPM cache（按 `Package.swift` hash）
  - `swift --version`（确认 runner 工具链）
  - `swift package resolve`
  - `swift build`（debug）+ `swift build -c release`
- **`release`**：`if: github.event_name == 'push' && github.ref == 'refs/heads/main'`，依赖 build
  - 重新 build release（缓存）
  - 内联组装 `.app`：复制二进制 + 写 Info.plist（含 `CFBundleDisplayName=Quota Bar`，与 `build-app.sh` 一致）
  - 打 .dmg：`hdiutil create -volname "Quota Bar" -srcfolder ... -format UDZO`，把 `.app` 放进去并加一个 `/Applications` 软链
  - 上传：[`softprops/action-gh-release@v2`](https://github.com/softprops/action-gh-release)，tag 格式 `nightly-<sha>`，`prerelease: true`

Runner：`macos-26`（项目最低系统版本就是 26；如果 runner 不可用会立刻报错，用户切到 `macos-15` + 自带 Xcode 26 即可）。

## 设计权衡

| 决策 | 选择 | 取舍 |
|---|---|---|
| Release 触发 | push main 自动建（不靠 tag） | 每次提交都产 release，列表会变长；优点是完全免手动 |
| Release 命名 | `nightly-<sha>` + `prerelease: true` | 不污染 "Latest Releases"；用户能在 Releases 页过滤出 nightly |
| Runner | `macos-26` | 项目最低系统版本 26.0；2026-06 应当已上线 |
| 脚本复用 | CI 不调 `build-app.sh`，inline 打包 | 脚本需要支持 `--release` 参数会有破坏性；CI 直接 inline 更稳 |
| `.app` vs `.dmg` | `.dmg` | 用户明确要求 dmg |
| DMG 格式 | UDZO（read-only + 压缩） | 文件小、传输快；mount 时是只读，更安全 |
| DMG 体积 | 只放 `.app` + `/Applications` 软链 | 经典 Mac 安装 UX：拖到软链即装 |
| 签名 / 公证 | 不做 | 第一次启动右键打开即可；要做需要 Apple Developer ID + signing secret，工作量大、用户没要 |
| 信息来源 | 单个 `release.yml` 同时管 PR check 和 release | 避免新增第二个 workflow 文件 |
