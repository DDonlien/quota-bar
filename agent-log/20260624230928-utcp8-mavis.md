### 1. 旧实现的语义偏差

**用户纠正**：上轮"整组 worst"的实现方向是对的（Kimi 1 组取 worst），但用户原话"基于排序取值其实都没做好"指的是**订阅组层面的排序**，不是 quota 层面的排序。

**重新对齐订阅组（subscription group）边界**：

| Provider | 订阅组数 | 订阅组边界 | 备注 |
|----------|---------|------------|------|
| Codex | 1 | "codex"（5h + 周共享额度） | parser 两个 window 共享同一组 |
| Kimi | 1 | "kimi"（Code 5h + Code 1w + Work 1m 共享额度） | 任一归零即全废 |
| MiniMax | N（按 modelName） | 每个 modelName（General / Video 等）= 1 个独立订阅，组内 5h + 周共享额度 | parser 按 modelName 分组 |
| Antigravity | N（按 group.displayName） | 每个 group.displayName = 1 个独立订阅（Gemini / Other / Claude 等） | 主路径按 group 分组；fallback 路径固定为 "Gemini" + "Other" |

**核心语义**：
- **bar / 灯取值 = 用户排序后第一个订阅组（top subscription group）里 remainingFraction 最小的 quota**。
- 单订阅组 provider（Codex / Kimi）等价于"整组 worst"。
- 多订阅组 provider（MiniMax / Antigravity）尊重订阅组排序。

### 2. 数据层改动

**`QuotaModels.swift`**

- `QuotaWindow.groupKey` → `QuotaWindow.subscriptionGroup`，注释升级为"独立计费的订阅组"语义。
- 字段 fallback 规则：`subscriptionGroup ?? providerKind.rawValue`（未显式设的 parser 视为整个 provider = 1 个订阅组）。
- 新增 `ProviderSnapshot.subscriptionGroups(customOrder:) -> [(groupKey: String, items: [QuotaWindow])]`：按 subscriptionGroup 分组，组顺序继承用户排序。
- 新增 `ProviderSnapshot.primarySubscriptionGroupWorstQuota(itemOrder:) -> QuotaWindow?`：取排序后第一个组里 worst quota。
- `statusColor(itemOrder:)` 内部切换到 `primarySubscriptionGroupWorstQuota`，注释强调"尊重用户排序"。
- 旧的 `worstQuota(itemOrder:)` 保留作 fallback API（忽略订阅组边界，直接取所有 quota 中最差）。

**各 Parser 显式设 subscriptionGroup**

| `PlaceholderProviders.swift` | 不显式设（fallback 到 providerKind.rawValue，单订阅组场景） |

**`PreferencesStore.swift`**

- `QuotaPreferences` 加 `subscriptionGroupOrder: [String: [String]]` 字段（key = providerKind.rawValue，value = 该 provider 下 subscriptionGroup key 的顺序）。
- 新增 `subscriptionGroupOrder(for:) -> [String]` 与 `setSubscriptionGroupOrder(_, for:)` 读写接口。
- Codable 向后兼容（旧配置缺该字段时 decoder 用默认值 `[:]`）。

### 3. UI 层改动

**`StatusBarController.swift`**

- `statusBarQuota(for:)` 改用 `primarySubscriptionGroupWorstQuota(itemOrder:)`，itemOrder 来源切换到 `subscriptionGroupOrder(for:)`。
- 关键改动：**bar 高度 / 状态灯取值与 quota 拖拽顺序解耦**，只跟订阅组顺序绑定。
- `observeCoordinator` 简化：菜单打开期间不再走 `needsRebuild` / `rebuildMenu`，由 NSHostingView 内的 SwiftUI 自己响应（见下）。

**`MenuView.swift`**

- `MenuView` 签名从 `state: DashboardState` 等 value params 改成 `@ObservedObject var coordinator: RefreshCoordinator`，computed props 转发 `state / isRefreshing / lastUpdatedText / needsFullDiskAccess`。
- 这是 dropdown 实时刷新的关键：SwiftUI 自动订阅 `@Published` 变化，菜单打开期间 `coordinator.state` 更新时 NSHostingView 内的 SwiftUI 树原地重渲染，不需要重建 NSMenuItem。
- `StatusBarController.rebuildMenu` 调用方相应更新（去掉 state/isRefreshing/lastUpdatedText/needsFullDiskAccess 显式传参）。

- `PlanSection` 重构：
  - 单订阅组 provider（Codex / Kimi / Claude / Cursor 等）：保持现有 UI（planHeader 带状态灯 + quota rows 平铺）。
  - 多订阅组 provider（MiniMax / Antigravity）：planHeader 不显示总状态灯（避免「多个状态灯打架」），改为渲染多个 `SubscriptionGroupBlock`。

- 新增 `SubscriptionGroupBlock`：
  - `SubscriptionGroupHeader`：订阅组名 + 该组状态灯（状态灯 = 该订阅组的 worst，颜色由 `statusColor(itemOrder: groupOrder)` 计算）。
  - quota rows（组内拖拽，scopedGroupKey 限定作用域）。
  - 订阅组本身可拖：payload = `"<kind>:<groupKey>"`，新增 `SubscriptionGroupDropDelegate` 只接受同 provider 内的 drop。

- `PlanHeader` 简化：根据订阅组数量决定是否显示状态灯（多订阅组时用占位保持缩进一致，避免 UI 抖动）。

- `QuotaRows` 接受 `quotas: [QuotaWindow]` 参数（不再从 `snapshot.orderedQuotas` 派生），让调用方传入已分好组的 quota list。

- `DraggableQuotaRow` / `QuotaItemDropDelegate` 加 `scopedGroupKey: String?` 参数：
  - `nil` → 单订阅组 provider，payload = `stableKey`，跨 provider 不限定；
  - 非 nil → 多订阅组 provider，payload = `"<kind>:<groupKey>:<stableKey>"`，`QuotaItemDropDelegate` 解析 payload 后只接受同组内的 drop（跨组拖拽被静默忽略）。

**dropdown 实时刷新验证**

- 之前：`MenuView` 接受 value-type `state`，NSHostingView 内的 SwiftUI 树不会响应 `coordinator.state` 变化，必须 `rebuildMenu` 重建 NSMenuItem 才能看到新数据——而 `rebuildMenu` 在 `isMenuOpen == true` 时只设 `needsRebuild`，关闭菜单才生效。所以用户必须开关 dropdown 才能看到新数据。
- 现在：`MenuView` 持有 `@ObservedObject var coordinator`，`coordinator.$state` 变化时 SwiftUI 触发 body 重算，NSHostingView 内部差异渲染，原地更新 UI。
