## 设计决策（先讲清楚再动手）

### 整体架构

- 使用 SwiftUI 标准 `Settings { ... }` scene，**自动**接入 `⌘,` 快捷键 + 应用菜单「偏好设置…」项 + Liquid Glass window 材质——避免自绘 NSWindow 重复造轮子
- `NavigationSplitView` 容器：sidebar (`listStyle(.sidebar)`) + detail (`Form { ... }.formStyle(.grouped)` + `.background(.regularMaterial)`)
- 4 个页面放在 `Sources/QuotaBar/Preferences/` 子目录，按"用户面向"语义拆分（不是按代码模块）

### sidebar 分组实现

按用户截图要求"默认组无标题 / Quota Bar 有标题"：

```swift
List(selection: $selection) {
    // 默认组：直接放 List 顶层，无 Section header
    ForEach(.allCases.filter { $0.group == .default }) { ... }

    // Quota Bar 组：用 Section { ... } header: { Text("Quota Bar") }
    Section { ... } header: { Text("Quota Bar") }
}
```

`PreferencesSection` enum 用 `.group: PreferencesGroup` 字段声明归属，sidebar 渲染时按 group 拆 filter——结构干净、可扩展。

### launchAtLogin 真假切换

`SMAppService.mainApp.register()` / `.unregister()` 需要有效 `.app` bundle；SwiftPM 直跑时 `Bundle.main.bundleIdentifier` 是 `org.swift-package-manager.quota-bar` 占位，会让 SMAppService 抛错。

处理方式：toggle 永远可点 + 永远持久化；调用 `SMAppService` 前先 `canRegisterLaunchItem` 检查 bundle id；不满足时 footer 文案提示"用 `./scripts/build-app.sh` 打包 .app 后再启用"。这样：
- SwiftPM 开发模式下 toggle 能切、状态会持久化，但 footer 解释为何系统级登录启动没生效
- 打包成 .app 后 toggle 自动真正注册

## 备注

- 本轮**未**给 PreferencesStore 写额外单元测试：Preferences 页面是 SwiftUI View，标准做法是用 SwiftUI Previews + UI 测试覆盖；单元测试留给后续 phase 在引入"设置偏好后行为"业务逻辑时再做（如"关闭 Provider X 后 RefreshCoordinator 是否真的过滤"）。
- "激活"页是占位骨架——激活体系本身是另一个独立 phase，本轮按用户截图把 4 个页面 UI 都做出来，激活页只展示状态 + 设备 ID + GitHub 引导。
- "关于"页用 `Bundle.main` 读版本/构建号；SwiftPM 直跑时 `CFBundleVersion` 是 nil（fallback 到 "dev"），打包成 .app 后会读到 build number。
- `Selector("showSettingsWindow:")` 用字符串字面量写法（`Selector(("showSettingsWindow:"))`）规避编译期 unknown selector 警告——这是 Swift 调用系统 selector 的标准做法。
- `deviceIDMasked` 用 `Bundle.main.bundlePath.hashValue` 派生占位 ID：同一进程稳定、不同 build 路径不同；激活体系上线后接入真实设备指纹（不在 UI 层直接读 `IOPlatformUUID` 等敏感硬件值）。
