## 对话与行动记录

### 方案设计

**核心取舍**：要不要按路由 (`/en` + `/zh`) 拆分页面 vs 单页客户端切换？

- 路由拆分：构建两份 HTML，部署两份路径（`/` 默认，`/zh/` 走前缀）；SEO 友好、URL 可分享；但 Astro 静态站需要双倍 `src/` 内容或动态路由。
- 客户端切换：单 HTML 双套文案，CSS 按 `data-locale` 切显示；构建简单、URL 简洁；代价是 SEO 默认一份（SSR 时给英文），中文 SEO 走 substring 匹配。

**决定走客户端切换**，原因：
1. 网站是单页 marketing page，SEO 不是核心诉求（不做内容营销）
2. Nav 里已经有占位的语言切换 UI（EN + ↓），明显是为客户端切换预留的
3. 切换体验更顺滑（同 URL 无刷新）
4. 部署只有一份 HTML，vercel 配置简单

**实现方式**：DOM 里同时渲染两套语言文案，CSS 按 `<html data-locale="..">` 切 `display:none`。

**默认 locale 检测顺序**（高优先 → 低）：
1. `localStorage.qb_locale`（用户手动切换过 → 锁定）
2. `navigator.language` / `navigator.languages`（zh-* → "zh"，其它 → "en"）
3. fallback "en"

**`navigator.language` 解析**：浏览器可能返回 `zh-CN`、`zh-TW`、`zh-HK`、`zh`、`en-US`、`en` 等；只看前缀 `zh` 就算中文（不分简繁：`zh-TW` 用户既看英文也能看简体中文，无歧义）。

**FOUC 防御**：`<head>` 同步 inline script 检测 + 设 `<html lang>` 和 `<html data-locale>` 属性，CSS 立刻生效；由于 SSR 输出的 DOM 同时含两套文案，浏览器拿到 HTML 后立即按默认 locale 显示，无任何闪烁。

**打字机 agent 列表**：两种语言都用同一组英文专有名词（Antigravity / Codex / Claude / MiniMax / Kimi / Zcode），但保留 `data-words-en` / `data-words-zh` 双属性以备未来扩展（比如本地化名称）。

### 关键 bug 修复（已修）

**症状**：第一次跑测试中文浏览器显示仍然英文。

**根因** 1：`JSON.stringify({ en, zh })` 在 Astro frontmatter 里序列化成字符串，注入浏览器后 `dictJson.en` 是 undefined（字符串没有 `en` 属性，访问得 undefined）。

**修法**：直接传对象 `define:vars={{ enDict, zhDict }}` → Astro 把它序列化成 `var enDict = {...}; var zhDict = {...};` 字面量。

**根因** 2（这是更深层的问题）：head inline 脚本里调 `apply(cur)` 想扫 `[data-i18n]` 替换文本，**但 head 脚本跑时 body 还没解析**，querySelectorAll 找到空集。

**修法**：把实际替换工作从 head 挪到 body 末尾（slot 之后、所有 layout 注入脚本之后），并让 head IIFE 通过 `window.QB_APPLY` 暴露函数。这样首次替换在 body 完整时执行 — 但仍在 first paint 之前（inline 脚本阻塞 parser），所以无 FOUC。
